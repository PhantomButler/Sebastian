Memory 统一写入与检索装配链路

对应 Spec：write-pipeline.md / retrieval.md / artifact-model.md · 对应代码：sebastian/memory/ · 2026-04-21

当前实现：Profile / Episode / Entity / Relation candidate 写入可用 动态 Slot：memory_slots 持久化 + 运行时热注册 Context / Relation 检索仍是轻量实现 Cross-Session / Full Maintenance deferred

0. 一句话总览

Sebastian Memory 不是把自然语言直接塞进一张表，也不是统一 search top-k 后丢给 LLM。 写入侧先把输入转成 CandidateArtifact，再通过 slot / subject / resolver 决定是否落库； 读取侧先判断当前用户消息意图，规划四条 lane 的检索预算，再按语义分区装配进 system prompt。

Artifact 是协议，不是表。 fact、preference、episode、summary、entity、relation 都先统一成候选产物，再由路由层分发。
Slot 是冲突域。 slot_id + cardinality + resolution_policy 决定新事实能否并存、取代、合并或丢弃。
LLM 只提议。 Extractor / Consolidator 不直接写库；数据库状态迁移由 Normalize / Resolve / Persist 控制。

1. Capture → Extract → Normalize → Resolve → Persist

2. 动态 Slot 注册

Slot 定义从代码常量升级为 memory_slots 持久化注册表。内置 9 个 seed slot 仍保留在 slots.py，启动时写入 DB，再从 DB bootstrap 到运行时 DEFAULT_SLOT_REGISTRY。 LLM 可以提出新 slot，但必须通过校验与注册流程后才会成为可用冲突域。

sequenceDiagram participant LLM as Extractor / Consolidator participant Tool as memory_save / Worker participant Handler as SlotProposalHandler participant DB as memory_slots participant Registry as DEFAULT_SLOT_REGISTRY participant Pipe as process_candidates LLM->>Tool: ExtractorOutput(artifacts, proposed_slots) Tool->>Handler: validate_proposed_slot(proposed) alt proposed slot 合法 Tool->>Pipe: candidates + proposed_slots Pipe->>Handler: register_or_reuse() Handler->>Registry: get(slot_id) alt registry 已有 Registry-->>Handler: 复用已有 SlotDefinition else registry 不存在 Handler->>DB: INSERT memory_slots inside savepoint alt INSERT 成功 Handler->>Registry: register(schema) else IntegrityError 并发冲突 Handler->>DB: get(slot_id) 读取赢家 Handler->>Registry: register(winner) end end Handler-->>Pipe: SlotDefinition Pipe->>Pipe: validate_candidate() 继续写入链路 else proposed slot 非法 Tool-->>LLM: 注入拒绝原因，最多重试 1 次 Pipe->>Pipe: 被拒 slot 对应 candidate 降级 slot_id=None Pipe->>Pipe: fact/preference 无 slot 将被 DISCARD end
命名规则：{scope}.{category}.{attribute} 三段式，纯小写 + 下划线，总长不超过 64。
字段组合：kind_constraints 不能为空；禁止 single + append_only；time_bound 至少适用于 fact 或 preference。
并发保护：register_or_reuse() 使用 savepoint；并发 INSERT 冲突时读取 DB 赢家并同步内存 registry。
启动恢复：seed_builtin_slots() 幂等写 seed；bootstrap_slot_registry() 加载 builtin + dynamic。

3. Intent → Retrieval Plan → Assemble

读取侧入口是 BaseAgent._memory_section()。它在每轮构造 system prompt 前调用 retrieve_memory_section()。当前 planner 是轻量规则：jieba 分词 + 词库交集 + Entity Registry 动态实体触发词。

Lane 语义职责 查询策略 为什么这样做
Profile 这个人长期是什么样：身份、偏好、稳定事实。 纯 SQL：status=active + 有效期过滤，按 confidence / recency 排序，默认 top 5。 画像抽象度高，字面 FTS 召回差；现阶段不引入 embedding，所以用小预算高置信注入。
Context 最近正在发生什么，尤其是当前项目或当前阶段。 Profile 表中的近期事实：jieba + FTS 优先，无命中时按 7 天窗口 fallback。 动态事实通常有较强字面相关性，适合 query-aware 检索。
Episode 过去发生过什么、上次讨论到哪里、历史决策依据。 Episode Store：先查 summary，summary 不够再查 episode detail。 summary 是高密度历史证据；detail 只在需要时补充，避免 prompt 噪声。
Relation 实体之间的关系，如用户、家人、项目、agent、工具。 Entity 触发词激活，查询 relation_candidates；实体名和 aliases 同步进 jieba 用户词典。 关系层仍在增强期，先保留候选层和检索挂点，后续可升级图查询。

4. Current Truth 与 Historical Evidence 的边界

Assembler 不把所有记忆压成一段自然语言摘要，而是保留分区和 kind 标签。这样 Agent 能知道哪些是当前有效事实， 哪些只是历史证据，避免把“上次发生过”误当成“现在仍然如此”。

自动注入门槛：检索硬阈值 confidence >= 0.3；自动注入路径额外要求 confidence >= 0.5。
policy_tags：do_not_auto_inject 阻止自动注入；access:* 和 agent:* 用于目的和 agent 类型过滤。
subject 隔离：Assembler 会二次检查 subject_id，防止跨主体串记忆。
