① 案例：用户偏好写入 → Session（会话）沉淀 → 下次对话自动注入 ✅

场景描述
Session（会话）1 · Turn（轮次）1：用户明确说"以后回复我都用中文，不用英文"
→ Sebastian 调用 memory_save 工具（Tool），写入 Profile Store（画像存储）中的 preference（偏好）slot（语义槽位）
→ 写入决策：ADD（新增，首次写入该 slot）

Session（会话）1 结束：SESSION_COMPLETED（会话完成）事件触发 Session Consolidation（会话沉淀）✅
→ SessionConsolidationWorker（会话沉淀任务）产出 summary（摘要），写入 Episode Store（经历存储）

Session（会话）2 · Turn（轮次）1：用户发新消息，语言偏好自动注入生效
→ _memory_section() 调用 Retrieval Planner（检索规划器）→ Profile Lane（画像通道）命中偏好
→ ⚠️ Intent（意图）判定当前为关键词规则，非 LLM 推断

时序图覆盖完整链路：即时写入路径（Explicit Write Path）✅、Session 后台沉淀（Session Consolidation）✅、 跨会话检索注入（Cross-session Retrieval Injection）✅ 三个阶段均已实现。

② 案例：偏好更新（SUPERSEDE · 取代）路径 ✅

场景描述
Turn（轮次）1：用户说"以后回复我都用中文"→ ADD（新增）写入 slot（语义槽位）
Turn（轮次）2（同一 session）：用户说"其实技术问题请用中英双语，其他问题还是中文"
→ Resolve（冲突解析）检测到同 slot 有 active（有效）记录 → 决策 SUPERSEDE（取代）
→ 旧记录 status → superseded（已取代），新记录写入，Decision Log（决策日志）记录 old_id / new_id 关联

展示 single-value slot（单值语义槽位）的覆盖更新路径：旧 Artifact（记忆产物）不删除而是标记为 superseded（已取代）， 保留历史轨迹，支持"为什么偏好变了"的审计查询。全路径已实现。

③ 案例：memory_search（记忆搜索）工具的 Lane-aware Budget（按通道分配预算）✅

场景描述
Agent 主动调用 memory_search(query="用户最近讨论的项目", limit=2)
→ Retrieval Planner（检索规划器）激活 3 条 Lane（检索通道）
→ effective_limit = max(2, 3) = 3（每条 Lane 至少 1 个名额）
→ Profile Lane（画像通道）1 条 + Context Lane（上下文通道）⚠️ 1 条 + Episode Lane（经历通道）1 条
→ Assembler（装配器）过滤后返回 3 条，含 citation_type（引用类型）字段区分来源语义

展示 memory_search 工具的 lane-aware budget（按通道分配预算）机制 ✅ 已实现。 effective_limit（实际上限）保证每条激活 Lane 至少 1 个候选名额，防止 Starvation（饿死）。 ⚠️ Context Lane 当前由 profile_store.search_recent_context() 提供，非独立 Context Store。

④ 已推迟路径（Deferred Paths）与待细化 Spec 项目

❌ 已推迟 / 未实现（Deferred / Not Implemented）

Cross-Session Consolidation
（跨会话沉淀）
当前只有 Session Consolidation（会话沉淀）由 SESSION_COMPLETED 事件触发。跨会话的偏好强化、模式归纳、多来源证据合并尚未实现。 需要单独 spec 定义：触发频率、扫描窗口、证据合并规则、幂等 key、Decision Log 记录方式和人工审核边界。
Full Memory Maintenance Worker
（完整记忆维护任务）
当前仅有 EXPIRE 生命周期动作（通过 write_router.persist_decision() 路由）。 尚未实现的：降权（Confidence Decay）、重复压缩（Dedup Compression）、索引修复（Index Repair）、独立周期性 Maintenance Worker 及其触发机制。
Exclusive Relation（互斥关系）
时间边界语义
relation_candidates（关系候选）写入已实现，但互斥关系（Exclusive Relation）的时间边界覆盖语义（新关系如何关闭旧关系 valid_until）尚未 spec 化和实现。
relation_facts（确认关系表）
晋升机制
relation_candidates 何时晋升为正式 relation_facts，以及 Relation Lane（关系通道）是否需要多跳查询（Multi-hop Query），均需单独 spec。
per-turn Conversational Inference
（逐轮对话推断）
"用户刚说完就能被下一轮检索到"的强实时能力。当前所有 Conversational Inference（对话推断）在会话结束后由 SessionConsolidationWorker 批量处理。 实时路径需要单独设计 debounce（防抖）、置信度阈值和撤销 / 审计策略。

⚠️ 部分实现 / 待细化 Spec（Partial / Needs Spec Refinement）

Intent 判定
（Keyword-based Rules）
当前实现为关键词词表匹配（EPISODE_LANE_KEYWORDS / RELATION_LANE_KEYWORDS 等小词表）。 Spec 描述为"轻量 LLM 或规则"。关键词覆盖不完整，边界模糊场景（如隐含的经历追问）可能激活失败。
Context Lane
（上下文通道）
当前由 profile_store.search_recent_context() 返回最近 active 的 Profile 记录，按时间排序。 Spec 描述为"query-aware（查询感知）动态事实检索 + FTS5 + Entity 命中"。当前实现无 FTS 检索，无独立 Context Store，与 Spec 存在差距。
Relation Lane
（关系通道）
当前从 relation_candidates（关系候选表）按 created_at 时间降序返回，无 query-aware FTS 检索。 entity_registry.list_relations() 仅按时间排序，实体名称匹配和关键词召回均未实现。
Summary Replacement
（摘要替换策略）
当前 episode summary（经历摘要）为 append-only（仅追加）。 何时替换已有摘要、何时追加新摘要、如何保留历史 evidence（证据），策略未 spec 化。 find_active_exact() 提供了精确去重，但无语义级替换策略。
Behavioral Observation
（行为观察）源 pipeline
MemorySource.observed 类型已定义，但首期无独立的行为观察采集 pipeline。 工具使用习惯、行为模式等目前暂由 SessionConsolidationWorker 从对话消息中隐式归纳，非专项处理。
