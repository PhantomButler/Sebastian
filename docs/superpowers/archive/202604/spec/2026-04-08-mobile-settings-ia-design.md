# Mobile Settings 信息架构重构设计

**日期**：2026-04-08
**状态**：Draft
**范围**：`ui/mobile/` 设置页信息架构与路由重组

## 1. 背景

当前移动端设置页将连接配置、Owner 登录、LLM Provider 管理、外观设置、Memory 入口和调试日志全部堆叠在一个长页面中：

- 首页同时承担导航页、状态概览页、编辑页三种职责
- 不同复杂度的设置项混排，视觉层级不清楚
- 随着 Provider 管理能力增加，设置页继续扩展会越来越拥挤
- 现有组件边界偏粗，首页直接承载了登录与 Provider 的重交互内容

这次重构的目标不是增加功能，而是重建设置页的信息架构，让用户先看到分类与状态，再进入对应页面完成修改。

## 2. 目标

- 将设置首页改造成按分类组织的状态面板，而不是长表单页
- 为不同设置类别建立清晰的二级路由
- 让每个详细设置页只处理本分类内容，不混入其他职责
- 保留少量高频快捷操作，但避免首页重新膨胀成操作台
- 为后续继续扩展设置项预留清晰边界

## 3. 非目标

- 不新增新的业务设置项
- 不修改后端 API、认证协议或 Provider 数据模型
- 不重做主题系统或 Provider 管理逻辑本身
- 不为了“层级完整”而引入没有信息增量的中间总览页

## 4. 用户体验结论

经过设计讨论，本次方案确认如下：

- 一级页采用“状态面板”风格，而不是传统紧凑列表
- 一级页负责展示分类摘要与当前状态
- 点击分类后直接进入该分类的详细设置页
- 不增加同名二级总览页，避免无意义多一跳
- 只有复杂对象编辑流才允许三级页存在
- 首页只保留极少数高频、低风险快捷操作

## 5. 信息架构

### 5.1 一级：设置首页

设置首页展示 4 张分类卡片：

1. 连接与账户
2. 模型与 Provider
3. 外观
4. 高级

每张卡片包含：

- 分类标题
- 一个主状态
- 一行摘要信息
- 点击进入详细页
- 如有必要，展示 1-2 个快捷操作

### 5.2 二级：分类详细页

首页点击分类后，直接进入对应详细页：

- `连接与账户详情页`
- `模型与 Provider 详情页`
- `外观详情页`
- `高级详情页`

二级页直接承载实际编辑内容，不再增加同名“总览页”。

### 5.3 三级：复杂对象编辑页

仅在对象本身具有列表与编辑流时使用三级页：

- `模型与 Provider`
  - Provider 列表页
  - Provider 新增页
  - Provider 编辑页

其余分类本次不拆三级，后续仅在内容体量明显增长时再扩展。

## 6. 各分类页内容范围

### 6.1 连接与账户

该页统一承载“接入 Sebastian”相关内容，包括：

- `Server URL`
- 当前连接状态
- `保存并测试`
- `Owner 登录`
- `退出登录`

归类原则：连接配置和身份接入都属于“能否连上并使用当前 Sebastian 实例”这一心智模型，应放在同一页。

### 6.2 模型与 Provider

该页承载与 LLM provider 相关的完整配置流：

- Provider 列表
- 默认项标识
- 空状态引导
- 新增 Provider
- 编辑 Provider
- 删除 Provider

这是当前设置系统中最复杂的一类，因此允许其使用三级编辑页。

空状态规则需要在本次明确：

- 当 provider 数量为 `0` 时，详细页展示空状态说明
- 空状态文案应明确告诉用户“尚未配置模型 Provider”
- 页面保留显著的 `添加 Provider` 主操作
- 首页卡片在空状态下不展示默认模型名，而是展示：
  - 主状态：`未配置`
  - 摘要：`尚未添加 Provider`

### 6.3 外观

该页本期承载：

- 跟随系统
- 浅色模式
- 深色模式

后续如增加字体、密度、动画偏好等视觉相关项，也统一归入本页。

### 6.4 高级

该页本期承载：

- `Memory` 入口
- 调试日志开关

后续任何偏开发、诊断、实验性或低频设置项，也统一归入本页。

其中 `Memory` 在本次迭代中的定义需要明确：

- `Memory` 仍为静态占位项，不新增真正的 Memory 详情页
- 该项在高级页中以“即将推出”样式展示
- 该项为不可点击项，不产生新路由
- 本次重构只调整它在信息架构中的归属位置，不扩展其功能范围

## 7. 首页卡片摘要设计

### 7.1 连接与账户卡片

- 主状态：`已连接` / `未测试` / `连接失败`
- 摘要：`10.0.2.2:8000 · Owner 已登录`

如果未登录，则摘要文案相应变为未登录状态。

这里的地址必须来自当前设置状态中的 `serverUrl`，不能硬编码为模拟器示例值。`10.0.2.2:8000` 仅用于说明摘要格式。

### 7.2 模型与 Provider 卡片

- 主状态：当前默认模型名
- 摘要：`provider_type · N 个 Provider`

其中 `provider_type` 指默认 Provider 的 `provider_type` 字段，不是 Provider 名称。

摘要同时表达默认来源与当前可选 provider 数量，帮助用户快速判断是否已完成模型配置。

空状态例外：

- 主状态：`未配置`
- 摘要：`尚未添加 Provider`

无默认项时的回退规则：

- 如果存在 Provider，但没有任何一项 `is_default = true`
- 主状态：`未设默认`
- 摘要：`N 个 Provider · 请选择默认项`

该规则必须显式实现，不能在首页隐式猜测第一项作为默认项。

### 7.3 外观卡片

- 主状态：`深色` / `浅色` / `跟随系统`
- 摘要：展示当前实际生效结果

目的：避免用户只看到“跟随系统”，却不知道当前界面实际呈现为浅色还是深色。

### 7.4 高级卡片

- 已登录时：
  - 主状态：`2 项设置`
  - 摘要：`Memory · 调试日志`
- 未登录时：
  - 主状态：`1 项设置`
  - 摘要：`Memory`

原因是当前 `DebugLogging` 仅在存在 `jwtToken` 时可见；首页摘要必须与详细页实际可见内容一致。

该卡片强调其为低频设置集合，而不是首页重点操作区域。

## 7.5 首页摘要状态矩阵

为避免首页文案在实现时被临时推断，本次明确以下状态矩阵：

### 模型与 Provider

- `providers.length === 0`
  - 主状态：`未配置`
  - 摘要：`尚未添加 Provider`
- `providers.length > 0` 且存在 `is_default = true`
  - 主状态：默认 Provider 的 `model`
  - 摘要：`${default.provider_type} · ${providers.length} 个 Provider`
- `providers.length > 0` 且不存在 `is_default = true`
  - 主状态：`未设默认`
  - 摘要：`${providers.length} 个 Provider · 请选择默认项`

### 高级

- `jwtToken` 存在
  - 主状态：`2 项设置`
  - 摘要：`Memory · 调试日志`
- `jwtToken` 不存在
  - 主状态：`1 项设置`
  - 摘要：`Memory`

## 8. 首页快捷操作规则

首页只保留高频且低风险动作：

- `连接与账户` 卡片可直接触发 `测试连接`
- `连接与账户` 卡片在已登录状态下可直接触发 `退出登录`

其余卡片不提供首页级直接编辑能力。

设计原则：

- 首页可以帮助用户快速确认系统是否可用
- 首页不能重新演化为“所有东西都能在这里改”的操作台

## 8.1 无 Provider 错误引导更新

当前移动端在未配置 LLM Provider 时，会在对话区域显示错误横幅并引导用户进入设置页。

本次设置页信息架构重构后，该引导必须同步更新：

- 当错误码为 `no_llm_provider` 时
- 对话区错误横幅的操作按钮不能再仅跳转到设置首页
- 必须直接跳转到新的 `模型与 Provider` 详细页，即 `/settings/providers`

目标是减少多一次导航决策，让用户直接到达可修复问题的页面。

适用范围：

- 主对话页 `app/index.tsx`
- Sub-Agent session 页 `app/subagents/session/[id].tsx`

这条规则仅针对 `no_llm_provider` 业务错误；其他如 `401` 导致的通用设置跳转不在本次变更范围内。

## 9. 路由设计

建议路由结构调整为：

```text
ui/mobile/app/settings/
├── index.tsx            # 设置首页：状态面板
├── connection.tsx       # 连接与账户详细页
├── appearance.tsx       # 外观详细页
├── advanced.tsx         # 高级详细页
└── providers/
    ├── index.tsx        # 模型与 Provider 详细页 / 列表页
    ├── new.tsx          # Provider 新增页
    └── [providerId].tsx # Provider 编辑页
```

本次直接采用独立新增路由约定，不使用特殊 id 兼容“新增”和“编辑”两种语义。这样实现计划可以明确区分：

- `providers/index.tsx`：列表与空状态
- `providers/new.tsx`：新增
- `providers/[providerId].tsx`：编辑

必须保持以下原则：

- 首页与分类页分离
- Provider 列表与 Provider 编辑流分离
- 不创建没有信息增量的中间页

## 10. 组件拆分原则

### 10.1 首页组件

首页组件只负责概览与入口，不承载复杂表单：

- `SettingsCategoryCard`
- 首页级摘要/状态展示组件
- 首页级快捷操作按钮

### 10.2 详细页组件

按分类拆分详细设置组件，例如：

- `ConnectionSettingsForm`
- `AccountSettingsSection`
- `ProviderListSection`
- `AppearanceSettingsSection`
- `AdvancedSettingsSection`

要求：

- 首页组件不直接持有 Provider 编辑表单
- 首页组件不直接持有登录输入表单
- 详细组件只处理本分类内容

## 11. 状态与数据流

本次重构不引入新的状态真相层，沿用现有移动端架构：

- `useSettingsStore` 继续负责本地配置状态与持久化
- 登录 / 登出继续走现有 `auth` API
- Provider 相关数据与交互继续基于当前逻辑迁移
- 首页只消费用于摘要展示的状态，不直接承载完整编辑逻辑

实现重点是“重组呈现方式与路由边界”，而不是重写业务逻辑。

## 12. 视觉方向

首页延续当前深色基调，但视觉角色需要明确变化：

- 首页卡片应表现为“状态入口卡”
- 详细页应回归更纯粹的设置列表 / 表单布局
- 首页不再展示半展开的长表单块

视觉预期：

- 首页更像面板式设置入口
- 详细页更像真正的设置编辑页
- 两者层级关系清晰，但不必完全同构

## 13. 测试与验证

本次改动至少应验证以下内容：

1. 首页 4 张卡片的状态摘要渲染正确
2. 首页点击后能进入正确的分类详细页
3. 首页快捷操作可用：
   - 测试连接
   - 退出登录
4. `连接与账户` 页中的连接与登录流程不回归
5. `模型与 Provider` 的新增、编辑、删除链路不回归
6. `模型与 Provider` 在空状态下展示正确文案，并可进入新增页
7. `外观` 设置修改后，首页摘要与实际主题同步
8. `高级` 页中的 Memory 占位项与调试日志开关可用

## 14. 风险与边界

### 14.1 复杂度集中风险

`模型与 Provider` 仍会是最复杂的设置类别。实现时应避免把首页逻辑耦合进 Provider 编辑流，否则会再次失去边界。

### 14.2 组件职责回流风险

如果继续把登录表单或 Provider 表单直接挂回首页，信息架构会很快退化回当前问题。

### 14.3 文案一致性风险

首页展示的是摘要状态，不是完整配置内容。实现时需要统一“主状态”和“副摘要”的生成规则，避免同一状态在首页和详情页表达不一致。

## 15. 文档同步要求

由于本次会调整移动端设置页路由结构与页面职责，实施时必须同步更新受影响的说明文档：

- `ui/mobile/README.md`
- `ui/mobile/src/components/settings/README.md`

如实现过程中调整了更细的设置页结构，也应同步检查上级索引是否需要更新，确保 README 导航与真实路由一致。

## 16. 结论

本次设置页重构采用“状态面板首页 + 直接进入分类详细页 + 仅复杂对象保留三级编辑流”的最短路径方案。

它解决的核心问题是：

- 首页不再承担所有编辑职责
- 设置项按用户心智分类
- 路由结构与页面职责对齐
- 后续新增设置项时有明确归属边界

该方案不追求一次性重做整个设置系统，而是在最短路径上完成信息架构重建，并为后续继续细化复杂设置页保留空间。
