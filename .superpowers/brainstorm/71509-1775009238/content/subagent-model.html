🤖 Sub-Agent 正确模型

有自主推理能力的"下属员工"，不是工具包装器

⚠️
修正之前的理解：MCP Server 只是暴露工具，没有推理能力。你要的 Sub-Agent 是有自己完整 Agent Loop（计划→执行→报告）的自治实体。 MCP 仍然有用，但用于 所有 Agent 调用外部工具，不是 Jarvis 和 Sub-Agent 之间的通信协议。
🔌 MCP（工具接入层）
Jarvis 和所有 Sub-Agent 都作为 MCP Client，通过 MCP 访问外部工具和资源。
MCP: 浏览器 MCP: 文件系统 MCP: 日历 MCP: 行情API
🤝 A2A（Agent 间通信层）
Jarvis 与 Sub-Agent 之间的协议：任务委派、进度上报、结果返回、升级请求。Google 已开放 A2A spec。
委派任务 进度上报 遇阻升级 结果交付

🏰 管家-仆人 类比映射
管家(Jarvis) → 统筹全局，接收主人指令，分配任务给下属，汇总结果
理财顾问 → StockAgent：主动分析，遇到不确定的事情才上报管家
工程师 → CodeAgent：接到需求自己搞定，做完告诉管家可以用了
生活管家 → LifeAgent：处理日常琐事，访客进门也由它负责接待判断
新仆人 → 新 Sub-Agent = 写 manifest 配置 + 可选新工具，系统自动纳编
⚡ 关键设计原则
下属自治 → Sub-Agent 默认自己解决问题，Jarvis 不微管理每一步
遇阻上报 → Sub-Agent 卡住才通过 A2A 升级，不是每步都问 Jarvis
能力共享 → 所有 Agent 共享基础工具集，专业化是叠加而非替换
并行协作 → 多个 Sub-Agent 同时工作，互相可通过 A2A 协作
渐进自主 → 工具/知识越丰富，需要上报的次数越来越少
💡 架构核心：三层协议栈
MCP（工具层）：所有 Agent 都用，接入外部工具和资源，语言无关。
A2A（Agent 通信层）：Jarvis ↔ Sub-Agent 之间的任务委派和结果交付协议。Google 已开放规范，可以直接采用。
内部 Agent Loop：每个 Agent 自己的推理-执行-上报循环，这是自主智能的核心，也是和 openjax 现有代码最直接相关的部分。

新增 Sub-Agent = 写一个 Agent Manifest（能力声明 + 专属工具 + 知识库配置 + 系统 Prompt），Jarvis 启动时扫描注册，A2A 自动可达。
